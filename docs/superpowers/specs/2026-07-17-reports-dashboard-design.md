## Overview

Sub-project #1 of the admin-UI structural refactor (see `docs/superpowers/specs/2026-07-17-admin-ui-structural-refactor-roadmap.md`, which this spec supersedes-in-detail for Reports specifically). Rebuilds `/reports` from a single flat DataTable of blocked-request rows into a tabbed dashboard matching the BunkerWeb Design System kit's `ui_kits/admin/pages/tables.js` `reports` mockup — Overview / Attack patterns / Top offenders — plus a 4th "Event log" tab that preserves the current table (the design kit has no equivalent for it; dropping it would be a regression).

**Design-kit source**: `.claude/BunkerWeb Design System/ui_kits/admin/pages/tables.js:647-936`.
**Current implementation**: `src/ui/app/templates/reports.html`, `src/ui/app/routes/reports.py`, `src/ui/app/static/js/pages/reports.js`, `src/api/app/routers/metrics.py`, `src/common/db/db_methods/metrics.py`, `src/common/db/model.py` (`Requests` / `bw_metrics_requests`).

## Goals

- Ship a real, data-backed Overview / Attack patterns / Top offenders tabbed dashboard using data BunkerWeb actually has (or can cheaply add — ASN).
- Preserve 100% of current Reports functionality (search, filter, sort, export, ban) inside a 4th "Event log" tab.
- Add a working time-range picker (1h/24h/7d/30d/custom) with period-over-period trend deltas, backed by new aggregation queries.

## Non-goals (explicitly out of scope, parked separately)

- **Compliance tab** (OWASP/PCI-DSS/HIPAA/GDPR scoring, audit dates, remediation backlog) — no backing data model anywhere in BunkerWeb; would be a net-new compliance-auditing feature, not a UI refactor. Parked as its own future chantier.
- **"Generate PDF"** whole-dashboard export — existing CSV/Excel server-side export (`reports.js:283-310`) stays as-is, unaffected. PDF needs a new rendering dependency; propose as a fast-follow.
- Literal "Investigating" / "MTTR" tiles — imply a manual incident-triage/acknowledgement workflow BunkerWeb doesn't have (blocking is automatic). Replaced with real derivable metrics (see Overview tab below), not fabricated placeholder values. Same reasoning as dropping Compliance.

## Data layer

### Schema change

`src/common/db/model.py`, `Requests` class (`bw_metrics_requests` table): add two nullable columns —

- `asn_number: Mapped[Optional[int]]`
- `asn_org: Mapped[Optional[str]] = mapped_column(String(256), nullable=True)`

Populated going forward only; existing rows render "N/A" for ASN in the UI. New Alembic migration required for all 4 engines (`mariadb_versions/`, `mysql_versions/`, `postgresql_versions/`, `sqlite_versions/`) — **auto-generate, do not hand-author** (per `[[alembic-migrations-autogenerated]]` house convention).

### Lua changes

- `src/bw/lua/bunkerweb/utils.lua:706` (`utils.get_asn`): currently returns only `result.autonomous_system_number`. Extend to also return `result.autonomous_system_organization` as a second value. ~~Additive — Lua multi-return means the 4 existing callers (`blacklist.lua:335`, `whitelist.lua:350`, `antibot.lua:1171`, `greylist.lua:266`) which only consume the first return value are unaffected.~~ **Erratum (post-Task-2 review, see `task-2-report.md` / `task-2-fix-report.md` / `task-2-review.md`): this claim is FALSE.** Inserting `asn_org` as the *2nd* return value (not appended last) shifts positional multi-return truncation, so all 4 callers' `local asn, err = get_asn(...)` silently rebind `err` to `asn_org` (`nil` on failure) and crash on `"string" .. nil` in their own error-logging branch on the normal ASN-lookup-failure path. Task 2 shipped with a follow-up fix changing all 4 callers to `local asn, _, err = get_asn(...)`. Any future task or reader relying on `utils.get_asn`'s return shape must account for this.
- `src/common/core/metrics/metrics.lua` (record builder, ~line 344-367): call `utils.get_asn(ctx.bw.remote_addr)` alongside the existing `get_country`/`get_reason` calls, add `asn_number`/`asn_org` to the record table sent to the batch-upsert endpoint.

### API / query layer

`src/common/db/db_methods/metrics.py` — new methods alongside `get_metrics_requests`/`get_metrics_facets`, following the same `_filter_conditions`/`_FACET_COLUMNS` pattern (report clause + search + facet filters, now also date-range bounded):

- **Time-bucketed counts**: `GROUP BY` a truncated `date` column (hour or day bucket per the `bucket` param), for the incidents-over-time chart. Also compute the same query over the immediately-preceding equal-length window, to derive trend-delta percentages (▲/▼).
- **Top offenders**: `GROUP BY ip` within the date range → count, `MIN(date)` (first seen), `MAX(date)` (last seen), plus `country`/`asn_number`/`asn_org` (take any row's value, they're per-IP-stable) and the most frequent `reason` as "top rule."
- **Top ModSecurity rules fired**: for rows where `reason = 'modsecurity'` in range, `data` is a JSON blob containing `{"ids": [...], "msgs": [...], ...}` (see `utils.get_reason`, `metrics.lua`) — unpack `data.ids` per row and tally occurrences in Python (no native JSON-array-unnest needed across all 4 supported engines; row counts here are bounded by the report clause + date range, not the full table).

`src/api/app/routers/metrics.py` — new endpoint `GET /metrics/requests/timeseries` (or extend the existing `/metrics/requests` with optional `start`/`end`/`bucket` query params if that's a smaller diff at implementation time — decide during planning) exposing the above.

**Important distinction from `home.py`'s existing `BW_INSTANCES_UTILS.get_home_aggregates()`**: that's a separate, Redis-backed, ephemeral streaming aggregation used by the Home dashboard's charts (bounded by Redis retention window). Reports' new aggregations query the **persisted** `bw_metrics_requests` DB table directly — this is the historical/audit record, correctness and completeness matter more than it does for Home's live-glance stats. Do not conflate the two or try to reuse `get_home_aggregates` for Reports.

## UI

### New shared component: `range-picker`

Per the original component-library plan (Phase 2), this was already identified as a needed-but-unbuilt component — built for real here. Design-kit source: `ui_kits/admin/components/range-picker.js`. Segmented `role="group"` button toggle (1h / 24h / 7d / 30d / custom with `aria-pressed` state), dispatching a bubbling `CustomEvent('change', {detail:{value}})`; `custom` reveals a flatpickr range input (flatpickr is already vendored and used elsewhere — `bans.js`, `pro.js`). New paired `static/js/components/range-picker.js`.

### Tab navigation

Reuse the existing `tabs` pattern already used by `profile.html` — no new tabs component needed. Four tabs: Overview, Attack patterns, Top offenders, Event log.

### Overview tab

- 4 KPI tiles: **Blocked requests** (count in range, was "Incidents"), **Blocked rate** (blocked / (blocked + detect-only) as %, was "Mitigated"), **Unique attacker IPs** (distinct IP count in range, replaces "Investigating"), **Peak hour** (busiest time bucket in range, replaces "MTTR"). All 4 are real, derivable from the new time-bucket/top-offender queries.
- Incidents-over-time bar chart — reuse existing `chart_area`/`chart_donut`-style ApexCharts component, wired to the range-picker.
- "Most recent incidents" card — top-N rows of existing report data (id/reason/type/service/source/action), reusing existing badge/table conventions from `reports.html`.
- "Top offender ASNs" bars — from the new top-offenders-by-ASN aggregation (grouping the top-offenders query by `asn_org` instead of `ip`).

### Attack patterns tab

- 3 attack-family tiles (SQL injection / RCE / Bot-scraper) with trend arrows from the period-comparison query, categorized by existing `reason`/rule-ID prefix conventions (CRS 942xxx = SQLi, 932xxx = RCE, Antibot = bot).
- "Top ModSecurity rules fired" table — real, from the new rule-ID tally query.
- Attack-family breakdown bars — grouped by `reason`.

### Top offenders tab

- 4 KPI tiles: unique offenders, repeat offenders (≥10 blocks in range), countries, ASNs — all from the top-offenders aggregation.
- Offenders table: IP / country / ASN / blocks / top rule / first seen / last seen / actions. "Ban selected" reuses the `ban_selected` DataTables button **already implemented** in `reports.js:322` — not new wiring.

### Event log tab (not in design kit — required to preserve current functionality)

The current `reports.html` table (search, `search_panes` facet filters, sort, CSV/Excel export, per-row "Full URL"/"Security Report Details" modals) moves here unchanged. This is the one tab whose positional-column DataTables coupling (`reports.js`'s ~14 `columnDefs` targets) must be preserved exactly per the existing per-page coupling precautions — no column reordering, no new nesting inside cells.

## Error handling / edge cases

- Rows predating the ASN migration: `asn_number`/`asn_org` are `NULL` → render "N/A", not a blank cell or error.
- Empty/zero-row date range (new instance, quiet period): tiles show 0, chart renders an empty state, trend deltas show "—" instead of a divide-by-zero percentage.
- `custom` range with an unset/invalid flatpickr selection: range-picker keeps the previously-active preset until a valid range is picked (mirrors the design kit's `onClose` gating).
