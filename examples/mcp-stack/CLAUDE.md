# BunkerWeb MCP Server - Development Guide

You have access to the BunkerWeb MCP server tools for managing BunkerWeb deployments. This document provides essential context about BunkerWeb and best practices for using the MCP tools.

## About BunkerWeb

BunkerWeb is a next-generation, open-source Web Application Firewall (WAF) that provides:
- **ModSecurity integration** - Industry-standard WAF engine with OWASP Core Rule Set
- **Antibot protection** - Challenge-based bot detection (captcha, JavaScript, reCAPTCHA)
- **Auto-configuration** - Automatic HTTPS, fail2ban-style IP blocking, and more
- **Multi-architecture** - Docker, Linux packages, Kubernetes, Swarm support
- **Plugin system** - Extensible via Python plugins for custom security logic

## BunkerWeb Architecture

### Core Components

1. **Instances** - Individual BunkerWeb nodes (nginx processes)
   - Each instance has a unique hostname
   - Instances can be started, stopped, reloaded
   - Configuration changes are automatically applied by BunkerWeb

2. **Services** - Virtual hosts/server blocks
   - Each service maps to one or more SERVER_NAME values
   - Services can be in "online" or "draft" state
   - Variables override global settings per-service

3. **Global Configuration** - Shared settings across all services
   - Security modules (ModSecurity, Antibot, BadBehavior, etc.)
   - Default values inherited by all services
   - Can be overridden at service level

4. **Custom Configs** - Configuration snippets
   - Types: `http`, `server-http`, `modsec`, `modsec-crs`, etc.
   - Can be global or service-specific
   - Used for custom nginx directives or ModSecurity rules

5. **Plugins** - Extend BunkerWeb functionality
   - Core plugins: antibot, bans, modsecurity, realip, etc.
   - External plugins can be uploaded as archives
   - Each plugin can schedule jobs

6. **Jobs** - Scheduled tasks run by plugins
   - Update threat intelligence feeds
   - Refresh IP blacklists/whitelists
   - Generate cache files used at runtime

7. **Cache** - Data files generated by jobs
   - IP lists, GeoIP databases, CRS rules
   - Stored per-plugin, per-job
   - Can be inspected or deleted via API

8. **Bans** - Dynamic IP blocking
   - Temporary or permanent bans
   - Can be service-specific or global
   - Includes metadata: reason, expiry, country

9. **Semantic Search** - AI-powered documentation search
   - FAISS-based vector search over BunkerWeb documentation
   - Multilingual support (English/French)
   - 2,621 optimized chunks for precise results
   - Automatic relevance filtering and deduplication

## MCP Resources Available

Reference these read-only resources in your conversations using the `@` syntax:

- `@config://global` - Current global BunkerWeb configuration
- `@logs://jobs` - Scheduler job execution history and status
- `@bans://active` - Currently active IP bans
- `@instances://status` - Health and status of all instances

Example: "Review @config://global and suggest security hardening improvements"

## Best Practices for BunkerWeb Operations

### Configuration Management

1. **Automatic configuration apply**
   - BunkerWeb automatically applies configuration changes
   - No manual reload is required after modifications
   - Use `reload_instances` with `test: true` only for syntax validation if needed

2. **Draft workflow for services**
   - Create new services in draft mode (`is_draft: true`)
   - Test thoroughly before converting to online
   - Use `convert_service` to promote draft → online

### Security Module Configuration

#### ModSecurity (modsec)
- **Anomaly scoring mode** (default) - Recommended for production
  - Scores requests based on rule matches
  - Block if anomaly score exceeds threshold
  - Less false positives than traditional blocking
- **Traditional mode** - Block on first rule match (stricter)
- **Important variables**:
  - `USE_MODSECURITY=yes` - Enable ModSecurity
  - `USE_MODSECURITY_CRS=yes` - Enable OWASP CRS
  - `MODSECURITY_CRS_ACTION=score` - Use anomaly scoring
  - `MODSECURITY_SEC_AUDIT_LOG=/var/log/nginx/modsec_audit.log` - Audit logging

#### Antibot
- Challenge types: `javascript`, `captcha`, `recaptcha`, `hcaptcha`, `turnstile`
- Session-based: bots must complete challenge before accessing site
- Variables:
  - `USE_ANTIBOT=javascript` - Enable JS challenge
  - `ANTIBOT_URI=/challenge` - Challenge endpoint
  - `ANTIBOT_SESSION_SECRET=<secret>` - Session signing key

#### Rate Limiting
- `USE_LIMIT_REQ=yes` - Enable request rate limiting
- `LIMIT_REQ_RATE=20r/s` - Requests per second per IP
- `LIMIT_REQ_BURST=40` - Burst allowance
- `LIMIT_REQ_CACHE=10m` - Memory for tracking IPs

### IP Ban Management

1. **Ban duration guidelines**
   - Temporary bans: 3600-86400 seconds (1-24 hours)
   - Permanent bans: `exp: 0`
   - Review long-lived bans periodically for accuracy

2. **Ban scope**
   - Global bans: `service: null` - affects all services
   - Service-specific: `service: "example.com"` - targeted blocking
   - Use service-specific when possible to minimize impact

3. **Audit and compliance**
   - Always provide meaningful `reason` field
   - Review active bans regularly with `list_bans`
   - Document ban decisions for security audits

### Job and Cache Management

1. **Job execution**
   - Most jobs run automatically on schedule
   - Manual runs: Use `jobs_run` for immediate updates
   - Check job history in `@logs://jobs` resource

2. **Cache maintenance**
   - Jobs populate cache files (IP lists, rules, etc.)
   - Stale caches may need manual refresh via `jobs_run`
   - Delete corrupt caches with `cache_delete_file`, then rerun job

3. **Common jobs**
   - `asn.download-asn` - Update ASN database
   - `blacklist.download-blacklist` - Refresh IP blacklists
   - `whitelist.download-whitelist` - Update trusted IPs
   - `modsecurity-crs.download-crs` - Update OWASP CRS rules

### Plugin Development

When creating or modifying plugins:

1. **Plugin structure**
   - `plugin.json` - Metadata and settings schema
   - `<name>.py` - Main plugin logic
   - Jobs defined in `plugin.json` under `jobs` array

2. **Settings inheritance**
   - Plugins define settings with defaults
   - Global config can override defaults
   - Service config can override global

3. **Testing plugins**
   - Upload via `plugins_upload` with `method: "ui"`
   - Test in isolated service before global deployment
   - Check logs for plugin initialization errors

## Common Workflows

### Deploy a new service

```
1. Create service in draft mode
   - Tool: create_service
   - Set is_draft: true, server_name: "example.com"

2. Add custom configuration if needed
   - Tool: config_create
   - Type: "server-http" for nginx directives

3. Convert service to online
   - Tool: convert_service with convert_to: "online"
   - BunkerWeb will automatically apply the changes
```

### Investigate security incident

```
1. Check recent bans
   - Resource: @bans://active
   - Look for patterns (same country, ASN, etc.)

2. Review ModSecurity audit logs
   - Access logs via file system or logging tool
   - File: /var/log/nginx/modsec_audit.log

3. Add preventive bans if needed
   - Tool: ban_ip
   - Set appropriate expiry and reason

4. Update WAF rules if vulnerability found
   - Tool: config_create (type: modsec-crs)
   - Add custom ModSecurity rules
```

### Update security rules

```
1. Check current job status
   - Resource: @logs://jobs
   - Identify last successful CRS update

2. Manually refresh rules
   - Tool: jobs_run
   - Jobs: [{"plugin": "modsecurity-crs", "name": "download-crs"}]

3. Verify cache updated
   - Tool: cache_list with plugin: "modsecurity-crs"
   - BunkerWeb will automatically apply the new rules
```

### Search documentation for configuration

```
1. Search for specific feature
   - Tool: search_bunkerweb_docs
   - Query: "how to configure ModSecurity with OWASP CRS"
   - Adjust min_score if too many/few results

2. Filter by category if needed
   - Use category parameter for targeted search
   - Example: category="features" for feature docs

3. Read full documentation at provided URLs
   - Each result includes direct link to docs.bunkerweb.io
   - Use URLs for complete context and examples
```

### Performance optimization

```
1. Review global configuration
   - Resource: @config://global
   - Check caching, compression settings

2. Adjust rate limits if needed
   - Tool: global_config_update
   - Increase LIMIT_REQ_RATE for high-traffic sites

3. Optimize ModSecurity
   - Disable unused CRS rule sets
   - Use MODSECURITY_CRS_ACTION=score for better performance

4. Monitor instance health
   - Resource: @instances://status
   - Check for overloaded instances
```

## Important Security Considerations

1. **Authentication**
   - MCP server supports Bearer tokens and HTTP Basic auth
   - Always use HTTPS in production for API communication
   - Rotate BUNKERWEB_API_TOKEN regularly

2. **ModSecurity tuning**
   - Monitor false positives in audit logs
   - Use paranoia level 1 initially, increase gradually
   - Whitelist known-good patterns via custom rules

3. **Antibot configuration**
   - JavaScript challenge: Good balance of security/UX
   - CAPTCHA: Higher security, worse UX
   - Exempt API endpoints from antibot challenges

4. **Regular maintenance**
   - Update CRS rules weekly or when CVEs published
   - Review ban lists monthly for expired/invalid entries
   - Test backups of global_config regularly

5. **Never commit secrets**
   - API tokens, session secrets, reCAPTCHA keys
   - Use environment variables or secrets management
   - Audit git history if secret accidentally committed

## Error Handling and Troubleshooting

### Common issues

1. **Configuration validation fails**
   - Check syntax in custom configs with `config_get`
   - Review nginx error logs
   - Validate ModSecurity rule syntax

2. **Service not responding after changes**
   - Check instance status: `list_instances`
   - Ping instance: `instance_ping`
   - Review SERVER_NAME matches DNS records

3. **ModSecurity blocking legitimate traffic**
   - Review audit log for rule IDs
   - Add exclusions via custom modsec config
   - Lower anomaly threshold or disable specific rules

4. **Jobs failing to execute**
   - Check `@logs://jobs` for error messages
   - Verify network access for download jobs
   - Ensure sufficient disk space for cache files

### Debug workflow

1. Use `ping` and `health` tools to verify API connectivity
2. Check `list_instances` to ensure instances are registered
3. Review `@config://global` for unexpected setting changes
4. Monitor `@logs://jobs` after configuration changes

## Semantic Search Tool

The `search_bunkerweb_docs` tool provides AI-powered semantic search across BunkerWeb documentation.

### When to Use Semantic Search

Use this tool when:
- User asks "how to" questions about BunkerWeb features
- Need to find specific configuration examples
- Looking for troubleshooting guidance
- Exploring security features and best practices
- Understanding plugin capabilities

### Search Parameters

- **query** (required): Natural language search query
  - Examples: "how to configure ModSecurity", "block malicious bots", "SSL certificate setup"
- **limit**: Maximum results to return (default: 5)
- **category**: Optional filter by category (e.g., "api", "plugins", "features", "integrations")
- **min_score**: Minimum relevance threshold 0-1 (default: 0.2)
  - 0.5+: Highly relevant results
  - 0.3-0.5: Good relevance
  - 0.2-0.3: Moderate relevance
  - <0.2: Low relevance (filtered out)
- **max_content_length**: Max characters per result (default: 500, 0=unlimited)

### Search Best Practices

1. **Use natural language queries** - The tool understands context and synonyms
   - Good: "how to configure ModSecurity WAF"
   - Good: "block malicious bots with JavaScript challenge"
   - Good: "comment bannir une adresse IP" (French supported)

2. **Adjust relevance threshold based on needs**
   - Exploratory search: `min_score=0.2` (broader results)
   - Precise answers: `min_score=0.4` (only high-quality matches)

3. **Use category filters for targeted search**
   - `category="api"` - API endpoints and usage
   - `category="features"` - Security features and configuration
   - `category="plugins"` - Plugin development and usage
   - `category="integrations"` - Integration guides (Docker, Kubernetes, etc.)

4. **Results are automatically deduplicated**
   - Only one chunk per document is returned
   - Always the most relevant chunk from each document

### Example Search Queries

```python
# Basic search
search_bunkerweb_docs(query="ModSecurity configuration")

# High-precision search
search_bunkerweb_docs(
    query="antibot JavaScript challenge configuration",
    min_score=0.4,
    limit=3
)

# Category-filtered search
search_bunkerweb_docs(
    query="API authentication",
    category="api",
    limit=5
)

# Multilingual search
search_bunkerweb_docs(query="configuration SSL automatique")
```

### Search Architecture

The semantic search system uses:
- **Model**: `paraphrase-multilingual-MiniLM-L12-v2` (384-dim embeddings)
- **Index**: FAISS (Facebook AI Similarity Search) with L2 distance
- **Chunking**: 320 characters per chunk with 64-character overlap
- **Documents**: 2,621 chunks covering all BunkerWeb documentation
- **Score calculation**: `1 / (1 + L2_distance / 10.0)` for 0-1 range

## When to Use Each Tool

### Read-only operations (safe)
- `ping`, `health` - Connectivity checks
- `list_*` tools - Inventory and status
- `get_*` tools - Detailed inspection
- `search_bunkerweb_docs` - Search documentation
- Resources via `@` syntax - Quick reference

### Modify operations (automatically applied)
- `create_*` tools - Adding new resources
- `update_*` tools - Changing existing resources
- `reload_*` tools - Manual reload (usually not needed, BunkerWeb auto-applies)
- `ban_ip` - Security enforcement

### Destructive operations (use carefully)
- `delete_*` tools - Permanent removal
- `unban_ip` - Removing security blocks
- `stop_*` tools - Service interruption

## Target API

The MCP server communicates with the BunkerWeb API at the base URL configured in `BUNKERWEB_BASE_URL`. This is typically:
- Development: `http://localhost:8888`
- Production: `https://bunkerweb-api.internal:8888`

All tool operations translate to REST API calls with retry logic and proper authentication.
